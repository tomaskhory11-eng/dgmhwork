# 北京公司办公自动化经营管理 AI Agent 设计规格

## 1. 背景与目标

本工程依据《北京公司办公自动化经营管理系统建设方案》建设。系统不按传统 OA 定位，而定位为“经营过程管理 AI Agent”：把客户、渠道、拜访、储备项目、合同、应收、回款、实施进度、计划汇报统一到一套可录入、可汇总、可分析、可部署的经营管理系统中。

第一版目标是做成本地可运行的网页应用，同时保留清晰后端和云服务器部署路径，后续可给多名同事协同使用。

核心效果：

- 个人按职责录入客户、渠道、拜访、项目、日报等基础动作。
- 系统自动汇总公司级项目池、合同回款状态、实施进度和经营指标。
- 支持拜访转储备项目、储备项目转合同、合同生成应收和实施进度。
- 自动生成周报草稿、月复盘草稿和经营分析提示。
- 本地先用 SQLite 快速落地，未来可迁移 PostgreSQL 上云。

## 2. 推荐技术路线

采用 `FastAPI + SQLite + 原生前端` 的一体化工程。

理由：

- 本地部署轻：Windows 上一键启动即可访问网页。
- 后端边界清楚：API、服务、数据模型分层，适合继续扩展。
- 云部署自然：FastAPI 可直接放到 Linux 云服务器，SQLite 可替换为 PostgreSQL。
- 维护成本低：第一版不引入重型前端构建链，便于非前端团队长期维护。

后续如进入正式多人云端使用，可升级为：

- 数据库：SQLite -> PostgreSQL。
- 登录：本地简化登录 -> JWT/Session + HTTPS。
- 部署：本地脚本 -> Docker Compose + Nginx 反向代理。

## 4. 用户角色与权限

第一版角色：

- 总经理：查看全部经营数据和看板；批注重点项目；分配项目；确认关键节点。
- 综合管理岗：维护合同、应收、回款、字段选项和管理报表。
- 商务人员：维护本人客户、渠道、拜访、储备项目、计划日报和回款跟进说明。
- 技术/实施人员：维护授权项目的实施节点、成果文件、延期说明和需协调事项。
- 系统管理员：维护账号、角色、字段字典、菜单权限和系统配置。

权限原则：

- 查看、修改、导出分离。
- 第一版用软删除/作废，不做物理删除。
- 关键字段修改写入操作日志。
- 本地单机阶段可以用简化登录；云部署前必须补齐密码哈希、会话过期、HTTPS 和权限校验测试。

## 5. 核心数据模型

第一版建立以下核心表：

- `users`：用户、角色、启用状态。
- `field_options`：客户大类、客户性质、业务板块、项目来源、项目阶段、付款方式等下拉字典。
- `customers`：客户/合作对象基础信息。
- `channels`：渠道、合伙人、机构资源。
- `visits`：客户拜访或有效沟通记录。
- `opportunities`：储备项目台账。
- `group_projects`：集团融合项目台账。
- `contracts`：合同台账，一份合同一行。
- `receivables`：应收款台账，记录应收计划和欠款状态。
- `payments`：回款明细，一笔到账一行。
- `delivery_projects`：实施项目进度台账。
- `plans`：月计划、周计划、日计划。
- `daily_reports`：日报。
- `generated_reports`：周报/月复盘草稿与生成记录。
- `audit_logs`：关键操作日志。

关键口径：

- 合同台账记录签约事实，不与应收台账混用。
- 回款明细用于核销应收，不反推签约额。
- 储备项目是经营漏斗核心，客户、渠道、拜访、合同都应尽量关联项目。
- 集团融合项目保留独立入口，不能隐藏在普通储备项目中。

## 6. 业务闭环与自动转化

第一版必须实现三类转化动作：

1. 拜访转储备项目
   - 在拜访记录中选择“是否形成商机=是”。
   - 系统根据客户、业务板块、预计金额、下一步动作生成储备项目草稿。
   - 用户确认后写入储备项目台账。

2. 储备项目转合同
   - 当储备项目阶段更新为“已签约”或点击“转合同”。
   - 系统生成合同台账草稿，带出客户、项目、业务板块、商务负责人、预计金额。
   - 合同编号、税率、直接成本、付款方式等由综合管理岗补齐。

3. 合同生成应收和实施进度
   - 合同确认后生成应收计划草稿和实施项目进度记录。
   - 应收计划由付款方式拆分，第一版可先支持手动编辑。
   - 实施项目带出合同编号、客户、项目名称、业务类型和项目负责人。

## 7. 页面与交互

第一版页面为一个本地网页工作台：

- 首页驾驶舱
  - 本月签约额、累计签约额、本月回款额、累计回款额。
  - 储备项目预计金额、重点项目数、逾期应收、延期实施项目。
  - 今日待办、回款提醒、需协调事项。

- 渠道客户
  - 客户基础信息。
  - 渠道合伙人台账。
  - 客户拜访台账。
  - 拜访转商机操作。

- 储备项目
  - 个人储备项目。
  - 公司项目汇总。
  - 集团融合项目。
  - 项目阶段管理和转合同操作。

- 合同回款
  - 合同台账。
  - 应收款台账。
  - 回款明细。
  - 合同到期、逾期天数、计划回款月份提醒。

- 项目实施
  - 实施项目进度台账。
  - 实施阶段、计划完成、实际完成、成果文件、客户确认、延期说明。

- 计划汇报
  - 月计划、周计划、日计划。
  - 日报录入。
  - 周报草稿。
  - 月复盘草稿。

- 系统设置
  - 用户角色。
  - 字段选项。
  - AI 配置。
  - 操作日志。

界面原则：

- 第一屏直接进入工作台，不做营销页。
- 业务系统风格应克制、密集、清晰，优先表格、筛选、状态、批量操作和可扫描指标。
- 日常录入字段要轻，能由系统带出的字段不让用户重复填写。

## 8. AI Agent 能力边界

第一版 AI 能力采用“规则生成 + 可选大模型增强”的双层设计。

本地规则生成必须可用：

- 周报草稿：从本周日报、拜访、渠道、储备项目、合同、回款、实施进度自动汇总。
- 月复盘草稿：从月计划、实际合同额、回款额、拜访数、渠道数、项目漏斗、交付进度生成结构化草稿。
- 项目跟进建议：基于项目阶段、最新跟进日期、下一步动作、计划完成日期给出待办提醒。
- 回款提醒：基于计划回款月份、计划回款金额、已收金额、逾期天数生成提醒。

大模型增强预留：

- 支持 OpenAI-compatible 配置：`base_url`、`api_key`、`model`。
- 未配置模型时，系统仍可用本地规则生成。
- 模型只生成经营分析、汇报润色、跟进建议，不直接修改关键台账。
- 模型输出必须保存为草稿，由用户确认后使用。

## 9. 后端 API 范围

第一版 API 包括：

- `GET /api/health`：健康检查。
- `GET /api/dashboard/summary`：首页经营指标。
- `GET/POST/PATCH /api/customers`：客户管理。
- `GET/POST/PATCH /api/channels`：渠道管理。
- `GET/POST/PATCH /api/visits`：拜访管理。
- `POST /api/visits/{id}/convert-to-opportunity`：拜访转储备项目。
- `GET/POST/PATCH /api/opportunities`：储备项目管理。
- `POST /api/opportunities/{id}/convert-to-contract`：储备项目转合同。
- `GET/POST/PATCH /api/group-projects`：集团融合项目。
- `GET/POST/PATCH /api/contracts`：合同管理。
- `POST /api/contracts/{id}/initialize-receivable-and-delivery`：合同初始化应收和实施。
- `GET/POST/PATCH /api/receivables`：应收管理。
- `GET/POST/PATCH /api/payments`：回款明细。
- `GET/POST/PATCH /api/delivery-projects`：实施进度。
- `GET/POST/PATCH /api/plans`：计划管理。
- `GET/POST/PATCH /api/daily-reports`：日报管理。
- `POST /api/reports/weekly/draft`：生成周报草稿。
- `POST /api/reports/monthly-review/draft`：生成月复盘草稿。
- `GET/POST/PATCH /api/settings/options`：字段选项。
- `GET/POST/PATCH /api/settings/ai`：AI 配置。

## 10. 数据导入与迁移

第一版先支持页面新增和编辑。随后增加 Excel 导入：

- 合同台账导入。
- 应收台账导入。
- 回款明细导入。
- 储备项目导入。
- 拜访台账导入。
- 字段选项导入。

导入策略：

- 先做字段映射预览，不直接写库。
- 对合同编号、项目名称、客户名称做重复提示。
- 导入错误生成可下载错误清单。
- 成功导入写操作日志。

## 11. 云部署预留

工程第一版即按云部署可迁移设计：

- 配置从环境变量读取，`data/local_settings.json` 仅用于本地开发。
- 数据库连接通过统一配置项读取。
- 文件路径不写死到 Windows 盘符，除本地启动脚本外使用相对路径。
- API 与前端分离，云端可由 Nginx 代理。
- 健康检查接口用于服务器监控。
- README 提供本地启动和云服务器部署步骤。

云部署推荐形态：

```text
Nginx -> Uvicorn/FastAPI -> PostgreSQL
```

可选 Docker Compose：

```text
services:
  web: FastAPI app
  db: PostgreSQL
  nginx: reverse proxy
```

## 13. 测试与验收

最低验收标准：

- 本地启动后可访问网页工作台。
- 主要台账可新增、编辑、作废、查询。
- 首页指标能根据台账数据变化自动更新。
- 拜访可生成储备项目。
- 储备项目可生成合同。
- 合同可生成应收计划和实施项目。
- 回款明细能影响应收已收/未收状态。
- 日报能汇总为周报草稿。
- 月计划和台账数据能生成月复盘草稿。
- 未配置 AI API 时系统仍可完整运行。
- 基础 API 测试通过。

## 15. 风险与约束

- 权限系统第一版不能过度复杂，但关键数据必须保留操作日志。
- 字段字典必须统一，否则统计会失真。
- 自动汇报先生成草稿，不直接代表最终经营判断。
- 大模型输出不能直接写入台账关键字段。
- SQLite 适合本地和早期试用；多人云端使用前应迁移 PostgreSQL。
- 当前 Word 方案给出了字段和流程，但没有提供完整历史 Excel 数据映射规则；Excel 导入需要在 MVP 后单独校准。
